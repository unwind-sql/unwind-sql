# Benchmark — `project.run` on the bundled example

Pipeline: 11 models (1 Python `raw_orders` + 10 SQL: 2 raw_*, 2 ref_*, 5 int_*
incl. one `@disabled`, 1 fct_*), DuckDB in-memory, sources read from parquets
generated by `example/generate_data.py` with **100 000 rows** of
`raw_orders`/`raw_shipments` and ~15 000 `raw_refunds`.

Target: the line called by `example/main.py`:

```python
project.run(vars={"d_reporting": "31/10/2025"}, debug=True)
```

Wrapped in [example/bench_run.py](example/bench_run.py) for hyperfine, plus an
in-process A/B in [example/bench_compare.py](example/bench_compare.py).

Hyperfine: `--warmup 2 --runs 8`, output suppressed to `/dev/null`.

---

## Optimization applied — `-- @materialized: view` directive

Before the change, every model was materialized as a DuckDB **table** by
[runner.py](src/unwind/runner.py): `CREATE OR REPLACE TABLE name AS (...)`.
Each step paid for one full scan of every parent + one full write of its own
output, and DuckDB's optimizer had **zero** visibility across `CREATE TABLE`
boundaries.

The new directive lets a SQL author opt models in or out:

```sql
-- @materialized: view
SELECT ...
```

Models flagged `view` become `CREATE OR REPLACE VIEW` — DuckDB inlines them at
query time, fuses scans, pushes filters/projections into the parquet readers,
and skips the intermediate write entirely. The default stays `table` for
backwards compat: nothing changes for a project that doesn't add the directive.

In the example, every SQL source (`raw_*`, `ref_*`), the Python `raw_orders`,
and every intermediate (`int_*`) is now a view. Only the final mart
`fct_warehouse_profitability` stays a table.

Implementation touches:

- [src/unwind/loader.py](src/unwind/loader.py) — parses `-- @materialized: view|table`.
- [src/unwind/project.py](src/unwind/project.py) — `Model.materialized` field.
- [src/unwind/runner.py](src/unwind/runner.py) — branches on the directive.
- [src/unwind/web/state.py](src/unwind/web/state.py) — same branch in the web layer.

## Wall-clock (hyperfine, includes Python + uv startup)

| Command | Mean [ms] | Min [ms] | Max [ms] | Relative |
|:--------|----------:|---------:|---------:|---------:|
| baseline (all tables) | 549.5 ± 29.1 | 507.2 | 592.5 | 1.00 |
| optimized (current mix) | 471.0 ± 28.9 | 431.9 | 523.7 | **1.17×** |

Wall-clock includes `uv run` resolving the env (~70 ms) and Python interpreter
startup + unwind/duckdb import (~200 ms). Those are constant across runs and
unaffected by the directive — they dilute the relative gain. The absolute
saving is `−78.5 ms`.

## Pipeline-only (in-process A/B, `RunResult.total_duration_s`)

`bench_compare.py` loads the project once per iteration but otherwise stays in
the same Python process; we measure only what `project.run(...)` reports.

```
n = 20 per config (after a 2-iteration warmup)

baseline (all tables)        median= 281.6 ms  mean= 278.8 ms  stdev= 26.7 ms  min= 241.2 ms  max= 332.0 ms
optimized (current mix)      median= 191.1 ms  mean= 189.9 ms  stdev= 16.4 ms  min= 161.1 ms  max= 225.9 ms

Median delta: -90.5 ms (1.47x speedup)
```

So the pipeline itself runs **~32% faster** end-to-end, with lower variance
(stdev 16 vs 27 ms). Once we strip the uniform Python startup, that's the real
win the directive buys.

## Why it works on this pipeline

`raw_orders` and `raw_shipments` are 100k rows of mostly numeric/string data;
the `int_*` chain joins them with the small `ref_*` tables and adds a couple
of derived columns each step. Replacing intermediates with views lets DuckDB:

1. Read each parquet **once** instead of once per dependent step.
2. Push the `qty > 0` filter from `int_order_base` all the way back to the
   `read_parquet` scan of `raw_orders` (skipping ~10 % of row groups).
3. Project away unused columns at the parquet boundary instead of carrying
   them through every intermediate table.
4. Fuse the chain of `int_order_base → int_transport_costs → int_tax_costs →
   int_net_margin_per_order` into a single physical plan.

The final aggregate `fct_warehouse_profitability` stays a table because (a)
it's the export target — we want it scannable for downstream UI/data tab uses
without recompute, and (b) it groups millions of values into 3 rows, so
materializing 3 rows is essentially free.

## Tradeoffs and when **not** to use `view`

- A view is **recomputed on every reference**. If two downstream marts both
  read `int_x`, marking `int_x` as view doubles its work. Keep it as table
  if it's read N times by independent consumers.
- The web UI's Data tab reads from the same DuckDB connection — viewing a
  100k-row view re-runs the upstream every page (no big deal at this scale,
  but worth noting). To keep browsing snappy on multi-million-row views,
  fall back to `table`.
- Views break the per-step row-count signal in `RunResult` slightly: we still
  report `row_count` (the runner runs `SELECT COUNT(*)` on the view), but
  this re-runs the whole upstream — a cost we accept since `debug=True`
  already pays for the per-step output.

## Generic, not example-specific

The directive itself doesn't assume any naming convention. Defaults are
backwards-compatible (`table` for any model without an explicit directive).
The example annotates `view` on its sources/intermediates as a demonstration;
real projects pick their own mix based on the consumer pattern of each model.
